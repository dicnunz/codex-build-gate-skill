# Venture Surgeon Operating Manual

Load this file for full reports, unfamiliar markets, live competitor checks, or when a verdict depends on current facts.

## Core Frame

A startup idea earns build time only when four things line up:

1. A specific person has a painful repeated problem.
2. That person already spends money, time, status, or attention trying to solve it.
3. The founder can reach enough of those people manually.
4. A 2-week test can create evidence that changes the decision.

If any one of those is absent, the output should shift from build planning to validation, pivoting, or killing.

## Evidence Quality Ladder

Rank evidence from strongest to weakest:

1. Paid commitment, signed LOI, deposit, pilot, or retained service.
2. Customer switches behavior or gives workflow/data access.
3. Customer repeatedly pays for a worse workaround.
4. Customer has an urgent deadline, compliance need, revenue leak, or visible operational cost.
5. Customer says the problem is painful in discovery and gives specific past behavior.
6. Customer joins a waitlist or says they would use it.
7. Founder thinks the market is large.

Use weak evidence only as a reason to run a test, not as proof to build.

## Scorecard Guidance

Use 1-5 scores tied to evidence:

| Area | 1 | 3 | 5 |
|---|---|---|---|
| Pain intensity | Nice-to-have | Annoying repeated pain | Costly, urgent, or embarrassing |
| Buyer clarity | Everyone | Named role/segment | Reachable buyer with budget |
| Urgency | No deadline | Seasonal or occasional | Immediate trigger or penalty |
| Current workaround | None | Manual workaround | Paid/currently tolerated bad tool |
| Differentiation | Feature clone | Sharper workflow | Clear wedge or unfair access |
| Distribution access | Cold generic audience | Known channels | Direct access to early adopters |
| Validation speed | Months | Several weeks | Testable inside 14 days |
| Founder edge | None | Some domain taste | Real access, proof, skill, or credibility |

Do not average scores mechanically. One fatal zero can dominate the verdict.

## Mode Playbooks

### triage

Output the decision, 3 strongest reasons, riskiest assumption, fastest test, and kill line.

### evidence-map

Build a table:

| Assumption | Evidence Now | What Would Prove It | What Would Kill It | Test |
|---|---|---|---|---|

Prefer tests that ask for money, workflow access, or behavior change.

### competition

Map:

- Current behavior: what users do today even if no product exists.
- Direct competitors: products sold as the same category.
- Indirect substitutes: spreadsheets, agencies, interns, scripts, manual process, ignoring the problem.
- Switching cost: migration, trust, habit, compliance, data, team process, price.
- Wedge: the narrow reason a first customer would switch now.

Browse when competitor existence, pricing, packaging, or recency matters.

### customer-route

Define:

- ICP: specific person, context, trigger.
- Source: where 30 of them can be found manually.
- Ask: one concrete behavior, not "feedback".
- Offer: paid/manual/small enough to buy now.
- Conversion threshold: how many replies, calls, payments, or workflow shares count as signal.

Avoid ads, launch platforms, generic communities, and growth hacks until manual reach works.

### mvp-contract

State:

- Core assumption: one sentence.
- Build: only the surface needed to test it.
- Concierge/manual parts: what the founder can fake by hand.
- Cut: every feature that does not test the assumption.
- Pass: numeric or behavioral threshold.
- Fail: what result means stop or pivot.

Default MVP window is 14 days.

### positioning

Find the wedge by completing:

- For [specific customer] who [trigger/problem],
- this helps them [concrete outcome],
- unlike [current behavior/competitor],
- because [credible edge or workflow difference].

Keep positioning grounded in proof. Do not create grand category language before traction.

### full

Use all of the following, compactly:

1. Decision.
2. Evidence ledger.
3. Scorecard.
4. Riskiest assumption.
5. Failure modes.
6. Competition/current behavior.
7. First 10 customer route.
8. MVP contract.
9. Kill criteria.

## Kill Criteria Examples

Use specific lines like:

- Kill if fewer than 3 of 20 target buyers agree to a paid/manual pilot.
- Kill if prospects describe the problem but already solve it well enough with a spreadsheet.
- Kill if the buyer is not the user and no budget owner responds.
- Kill if switching requires migration/trust work the wedge does not overcome.
- Pivot if non-buyers love it but one adjacent role has a sharper paid pain.

## Outreach Angle Rules

When asked for outreach, write one short message. It should:

- mention the exact role or trigger,
- ask about past behavior,
- offer a small concrete outcome,
- include a paid or behavior-changing ask when appropriate,
- avoid hype, flattery, and long context.

Do not include outreach by default unless requested.
